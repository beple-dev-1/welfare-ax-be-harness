# Failure Modes & Final Checklist

> dev-planner 산출물(루트/페이즈 문서·메타 JSON) Write 완료 후 자체 점검 시 단일 출처.

## Failure Modes To Avoid

- **브리프 재해석**: 브리프 §2에서 "개발 유형: 신규"로 결정된 것을 "사실 확장일 수도"라며 뒤집는다 → 브리프는 계약이므로 존중. 의견이 있으면 §4 전제에 가정으로 기록만.
- **브리프 섹션 누락 시 중단**: 브리프에 특정 §N(예: §4 업무↔코드 매핑)이 비어있거나 부재할 때 에이전트 실행을 중단한다 → **중단하지 않는다.** 해당 섹션에 대응하는 플랜 섹션을 `| — | 해당 없음 (브리프 §N 미작성) | — |` 단일 행으로 기록하고, 루트 §4 전제·가정에 "브리프 §N 보강 후 재계획 필요" 항목을 추가한다. 브리프 §3-4(영향 테이블)와 §11(구현 순서)이 모두 비어있는 극단 케이스만 실행 중단(사용자에게 dev-interview 재수행 안내).
- **scope 별도 요구**: 사용자에게 scope 식별자를 물어본다 → 브리프 §2·§3에서 자동 역매칭해야 한다.
- **페이즈 과다 분할**: §11 모든 단계를 1:1 페이즈로 → 휴리스틱 R4/R5/R7 적용.
- **페이즈 과소 분할**: 10단계 이상의 대형 브리프를 1페이즈로만 묶거나 페이즈당 4d 초과 → **R7**(개수 권장 4~7) 또는 **R6**(페이즈당 1~3d 권장) 위반. 브리프 규모 재평가 후 재분할.
- **페이즈 문서 간 강한 의존**: 페이즈 B가 페이즈 A의 상세 섹션을 끊임없이 인용 → 루트의 §6/§7/§8(글로벌)로 몰아내고, 페이즈 문서는 "루트 §N 참조"로 간결화.
- **Task 추상화**: "Controller 구현" 같은 막연한 Task → "POST /ceremony/apply 엔드포인트 추가 (0.5d)"처럼 구체화.
- **dev-plan 페이즈에 §DoD 또는 §검증 방법 섹션 작성**: DoD 폐기 정책에 따라 dev-plan 페이즈 문서는 §DoD·§검증 방법 섹션 모두 만들지 않는다. 페이즈 종료 신호는 BE = `./gradlew :{module}:test` GREEN, FE = sub-agent 코드 생성 완료 보고. 테스트·회귀·커버리지·통과 기준은 qa-plan 페이즈별 테스트 문서로 전부 위임.
- **설정 파일 읽기**: YML/Properties 열람 — 절대 금지.
- **실제 데이터 조회**: PostgreSQL `SELECT * FROM ...` — 메타데이터만.
- **메타 JSON 레거시 키 누락**: `work_request` 별칭 빠뜨리면 레거시 키만 읽는 qa-planner 가 침묵 → 반드시 이중 키 유지.
- **부실한 분할 근거**: 루트 §5-2에 어떤 규칙이 어떻게 적용됐는지 기록 안 함 → 사용자가 재그룹 요청 시 어려움.
- **검토 게이트 생략**: 4단계 페이즈 분할 직후 5~7단계로 직진 → 4.5단계 사용자 게이트 필수 (단, 초소형 과업 자명 케이스는 예외).
- **§4 코드 본문 작성**: 페이즈 문서 §4에 메서드 본문/SQL 본문/JS 함수 본문까지 작성 → `/develop` 영역 침범. 시그니처·책임·의존까지만.
- **4.5단계 검토 게이트 우회**: 페이즈 분할 검토 단계를 형식적으로 건너뛰고 자동 디스패치 흐름에만 의존 → 잘못된 분할이 그대로 qa-planner 진입. **4.5단계는 qa-planner 자동 디스패치 직전 사용자 합의 게이트** — 우회 금지. 분할이 자명한 초소형 과업만 4.5단계 자동 생략 허용.
- **R9 무조건 적용**: REST API / Batch / Library 같은 화면 없는 프로젝트에 R9를 억지로 적용해 "화면" 컬럼에 가짜 항목 채우기 → R9는 **프로젝트 유형 의존**. 화면 없는 프로젝트는 R9 미적용을 명시하고 R1~R8만 사용.
- **R9 vs R3 충돌 시 R9 우선 적용**: 신규 도메인이라도 R9의 화면 단위가 우선이라고 오해하여 신규 도메인 화면을 기존 도메인 페이즈에 묶음 → **R3가 R9보다 우선**. 신규 도메인은 화면 단위와 무관하게 독립 페이즈. (예: 부가 도메인은 신규 도메인이므로 등록 화면 페이즈에 묶지 않음.)
- **화면 1:1 미준수 (그룹핑·마법사 분할)**: R9 적용 시 *"마법사 3단계는 1 페이즈로 묶기"*, *"CRUD 목록+상세+등록을 1 페이즈로"* 같이 화면을 묶거나, 반대로 *"마법사 1개를 단계별 N 페이즈로 분할"* → **화면 1개 = 1페이즈** 단일 원칙 위반. 페이즈 = 사용자 검수 단위이므로 화면을 묶으면 검수 분리가 깨지고, 화면 1개를 N개로 쪼개면 검수 단위가 흩어진다. 화면이 너무 크다면 화면 자체를 다시 정의해야 하지 페이즈 분할 옵션이 아님 (UI 흐름 분할은 dev-plan 영역 밖).
- **테스트 페이즈를 dev-plan에 둠**: 마지막 페이즈를 "테스트 & DoD"로 만들고 §3·§4에 테스트 클래스명·시나리오 명세 → **qa-plan 영역 침범**. R5 보정에 따라 테스트 페이즈는 만들지 않는다. 상세 테스트 계획은 qa-plan 페이즈별 테스트 문서가 전담.
- **8섹션 구조 위반 (§DoD·§검증 방법 부활)**: DoD 폐기 정책에 따라 dev-plan 페이즈는 §1~§7 + §8(인계) **8섹션 구조**다. §DoD 또는 §검증 방법 섹션을 부활시키면 *코드 작성 완료 = 페이즈 종료* 신호가 흐려지고 qa-plan과 영역 중복. 페이즈 종료 신호는 BE = `./gradlew :{module}:test` GREEN, FE = sub-agent 코드 생성 완료 보고로 갈음한다.
- **풀스택 프로젝트에서 BE+FE 한 페이즈에 혼재**: 한 페이즈에 Controller·Service(BE)와 Template·JS(FE)를 함께 두면 → 컨텍스트 스위칭 비효율 + API 변경 시 FE 재작업 위험 + 페이즈 단위가 BE 작업 흐름과 FE 화면 단위 사이에서 모호해짐. **R10 적용 시 BE/FE 영역 분리 필수**. R9는 R10 채택 시 FE 페이즈 분할 축으로만 적용.
- **단일 BE 페이즈 슬러그를 영역 명으로 한정**: 단일 BE 페이즈에 `be-api` / `be-service` 슬러그를 사용하여 마치 일부 영역만 다루는 것처럼 보임 → 단일 BE 페이즈 슬러그는 `be`. BE 다단계 분할 시에만 `be-{레이어}` 또는 `be-infra` + `be-{도메인}` 사용.
- **BE 다단계 축 선택 누락**: BE 다단계 분할 시 어떤 축(레이어 축 vs 도메인 축)을 선택했는지 §5-2 분할 근거에 명시하지 않음 → 사용자가 "왜 이렇게 나눴는지" 추론해야 함. **분할 근거에 축 선택과 이유**(인프라 비중·도메인 수·도메인 독립성)를 1줄로 기록.
- **도메인 ≥ 2개 묶임 → 과업 분리 검토 없이 도메인 축 즉시 적용**: 한 과업에 도메인 ≥ 2개가 묶여 들어왔는데 4.5단계에서 사용자에게 *"과업 분리 가능한가?"* 확인 없이 도메인 축으로 바로 분할 → 운영 표준(도메인별 과업 분리)을 우회. **도메인 ≥ 2개 묶임은 4.5단계 선결 검토 의무** — 사용자에게 분리 가능 여부 확인 후, 분리 가능하면 dev-interview 재요청 권유. 분리 불가 판정 후에만 도메인 축 적용 (예외 케이스 한정).
- **레이어 축 강제 적용 (도메인 ≥ 2개 + 독립성 높은 케이스 + 분리 불가)**: 분리 불가 확정된 도메인이 여럿이고 독립성이 높은데 `be-infra` / `be-service` / `be-api`로 강제 분할 → `be-service` / `be-api` 페이즈에 두 도메인 코드가 섞여 검수 단위가 모호해짐. **도메인 ≥ 2개 + 독립성 높음 + 분리 불가 → 도메인 축**(`be-infra` + `be-{도메인}` × N) 안전망 적용.
- **도메인 축 강제 적용 (인프라 큼 + 도메인 1개)**: 인프라 비중이 크고 도메인이 1개인데 `be-{도메인}` 단일 페이즈로 묶음 → 인프라 작업이 도메인 페이즈에 끼어들어 검수 단위 모호. **인프라 ≥ 1d + 도메인 1개 → 레이어 축**(`be-infra` / `be-service` / `be-api`). 도메인 축은 도메인 ≥ 2개 분리 불가 케이스에만.
- **두 축 혼재 (한 과업에 레이어 축 + 도메인 축 동시 사용)**: 한 과업의 BE 다단계 페이즈에서 Phase 2 = `be-service`, Phase 3 = `be-payment`처럼 두 축을 섞음 → 일관성 깨짐. **한 과업은 한 축으로 통일** (단, `be-infra`는 두 축 공통 진입점이라 도메인 축에서 `be-infra` + `be-{도메인}` × N 조합은 정상).
- **메타 JSON 응답 텍스트만 작성 (실제 Write 누락)**: 응답 텍스트에 `.claude/tmp/dev-plan-meta.json` 경로를 명시하면서 실제 Write 도구를 호출하지 않음 → `/dev-plan` 스킬이 메타를 못 읽어 fallback(taskNumber 재구성) 경로로 떨어짐 (scope 등 메타 전용 정보 유실 위험). 8단계는 응답 텍스트가 아니라 **실제 Bash mkdir + Write 도구 호출**로 완료. 응답 직전 Bash `ls`로 작성 검증 권장.
- **결과 반환 옛 표현 잔존 (*"다음 단계 (사용자 명시 호출)"*)**: 응답 자체에 *"사용자 명시 호출"* / *"비활성화 권장"* / *"자동 실행 비활성화"* 같은 옛 패턴이 등장하면 → 사용자가 자동 디스패치 흐름을 인지 못 함. **9단계 결과 반환 포맷을 그대로 사용**, 자체 변형 금지.
- **§6-3 DTO 필드 명세 누락**: 루트 §6-3 에 DTO 명만 적고 필드 표를 빠뜨리거나, *"필드는 dev-backend·dev-frontend 가 결정"* 처럼 위임 → **시나리오 C 전제 파괴**. BE·FE 두 sub-agent 가 *서로 코드 안 보고* 같은 계약을 신뢰하는 것이 시나리오 C 핵심. 필드명·타입·검증 규칙이 dev-planner 시점에 결정되지 않으면 BE/FE drift 발생 → 기능 검증 단계 redo. 모든 Request·Response DTO 별 5컬럼 표(필드명·타입·필수·검증·의미) 의무.
- **페이즈 §4 DTO 필드 표 누락**: 루트 §6-3 은 채웠으나 페이즈 문서 §4-X 에 Request/Response DTO 필드 표(BE 페이즈) 또는 화면 input 필드 표(FE 페이즈)가 없음 → 페이즈만 보고 작업하는 sub-agent 가 루트 문서까지 참조해야 하는 부담 + 정합 깨질 위험. **페이즈 자기완결 원칙 위반**. 페이즈 문서 §4 안에 필드 표 의무.
- **BE/FE 필드명·검증 규칙 불일치**: 루트 §6-3 Request DTO 필드명 `userNm` 인데 FE 페이즈 §4-N input name 이 `userName` 또는 검증 정규식이 양쪽 다름 → AJAX payload key mismatch / 한쪽만 통과하는 검증 → 기능 검증 redo. **dev-planner 가 §6-3 작성 후 페이즈 §4-X·§4-N 작성 시 grep 자가 검증 의무** (필드명 동일 / 검증 규칙 BE ≥ FE).
- **FE 가 BE 보다 강한 검증 가짐**: FE input pattern 이 BE `@Pattern` 보다 더 좁은 정규식 → BE 가 받아들이는 합법 입력이 FE 에서 막힘. **BE 가 최종 게이트** 원칙 위반. FE ≤ BE 검증 강도 (FE 약함 OK, FE 강함 금지).

## Final Checklist

- [ ] 브리프 11섹션을 모두 파싱하여 플랜에 전이했는가?
- [ ] scope.yaml 역매칭으로 scope 정보를 자동 결정했는가?
- [ ] 신규 프로젝트인 경우 유사 유형의 기존 프로젝트 1~2개를 §3-3 패턴 참조에 명시했는가?
- [ ] 페이즈 분할이 R7 범위(권장 4~7, 허용 2~10) 내이고, 휴리스틱 근거가 루트 §5-2에 기록되었는가?
- [ ] 각 페이즈 문서가 **8섹션** 구조(§1~§7 + §8 인계)를 채우고, **§DoD·§검증 방법 섹션을 만들지 않았는가**? (DoD 폐기 — qa-plan 위임 + sub-agent `./gradlew :{module}:test` GREEN 신호)
- [ ] Task가 기능 단위 + 시간 추정 + 수락 기준을 모두 가지는가?
- [ ] 루트 문서·페이즈 문서가 각각 올바른 경로에 저장되었는가?
- [ ] 메타 JSON이 이중 키(dev_brief + work_request) + phase_files 배열을 담았는가?
- [ ] **메타 JSON을 응답 텍스트에 경로만 명시한 게 아니라 Write 도구로 실제 파일을 작성했는가?** (응답 직전 Bash `ls .claude/tmp/dev-plan-meta.json`으로 자체 검증 권장 — 파일 부재 시 스킬 fallback 경로로 전환)
- [ ] 결과 반환에 *"사용자 명시 호출"* 같은 옛 표현 대신 *"스킬 자동 디스패치"* 흐름을 명시했는가?
- [ ] 4.5단계 페이즈 분할 검토 게이트를 거쳤는가? (초소형 과업 자명 케이스는 생략 사실 명시)
- [ ] §4 파일별 구현 상세가 시그니처·책임·의존 수준이고, 메서드 본문은 작성하지 않았는가?
- [ ] 결과 반환 안내가 qa-planner **스킬 자동 디스패치 흐름**(4.5단계 분할 검토 통과 → /dev-plan 스킬이 메타 JSON Read → qa-planner 호출 → qa-planner 6.5단계)을 명시하는가?
