# Phase Partitioning Examples

> 4.5단계 검토 게이트 및 분할 의사결정 시 참고할 6개 실 사례.

---

**예시 1 — 풀스택 대형 과업 (R9 + R10 동시 적용, BE 레이어 축 — Spring Boot Web / 화면 다수, 인프라 비중 큼 + 도메인 1개):**

브리프 §11 15단계 + §7 화면 맵(로그인·관리자·등록 v1/v2·외부 SSO·부가 도메인) → **R9 + R10 + 레이어 축 적용** → **BE 다단계(3, 인프라→Service→API) + FE 화면별(4) = 7 페이즈**:

| N| 페이즈명| 영역| slug| §11 단계| 포함 화면| 적용 규칙| 기간|
|---|---------|------|------|---------|---------|----------|------|
| 1| BE 인프라 & 공통 & 보안| BE| be-infra| 1,2,3| (없음)| R1 + R10 BE 다단계(레이어 축)| 2.5d|
| 2| BE 도메인 Service| BE| be-service| 4,5,6,8,9,10| (없음)| R10 레이어 축| 2.5d|
| 3| BE Controller & API| BE| be-api| 7,11| (없음)| R10 레이어 축| 1.5d|
| 4| FE 로그인| FE| fe-login| (4 BE 후)| login.html, otp-modal| R9 + R10 + R4(Account 흡수)| 1.5d|
| 5| FE 등록 v1/v2 + SSO| FE| fe-entry| (BE 후)| entry/v1, entry/v2, external-link| R9 + R10 + R2(v1/v2 분리)| 4d|
| 6| FE 관리자| FE| fe-admin| (BE 후)| admin/list, admin/stat| R9 + R10 + R4(History 흡수)| 2d|
| 7| FE 부가 도메인| FE| fe-addon| 12,13,14,15| addon/*| R9 + R10 + R3(신규 도메인)| 3d|

> **레이어 축 선택 이유**: 도메인이 1개(통합 도메인)이면서 인프라(보안·공통 설정·횡단 모듈)가 2.5d 비중을 차지 → 레이어 축이 자연. BE 페이즈 검수 = `be-api` 도달 시 진짜 동작 확인.

**예시 2 — 풀스택 대형 과업** ️ **비권장 시나리오 (운영상 드문 케이스)** — 한 과업에 도메인 ≥ 2개가 묶여 들어와 분리 불가한 예외 케이스 한정 패턴 시연. 운영 표준은 **도메인별 과업 분리** → 분리 가능하면 각 과업당 도메인 1개 = 레이어 축 또는 단일 BE로 자연 처리. 분리 불가 판정 후에만 이 패턴 적용 (R9 + R10 + 도메인 축, Web MVC / 화면 다수, 도메인 ≥ 2개 + 독립성 높음, 분리 불가):

> **선결 검토 필수**: 4.5단계 게이트에서 사용자에게 *"도메인 ≥ 2개가 묶여 있는데 과업 분리 가능한가?"* 확인. 분리 가능하면 즉시 도메인 1개씩으로 과업 분리 후 dev-interview 재요청 권유. 분리 불가일 때만 아래 6페이즈 패턴 진행.

브리프 §11 12단계 + §7 화면 맵(결제·가맹점·정산 도메인 동시 추가, 분리 불가 확정) → **R9 + R10 + 도메인 축 적용 (예외 케이스)** → **BE 다단계(3, 인프라+도메인×2) + FE 화면별(3) = 6 페이즈**:

| N| 페이즈명| 영역| slug| §11 단계| 포함 화면| 적용 규칙| 기간|
|---|---------|------|------|---------|---------|----------|------|
| 1| BE 인프라 & 공통| BE| be-infra| 1,2,3| (없음)| R1 + R10 BE 다단계(도메인 축)| 1.5d|
| 2| BE 결제 도메인| BE| be-payment| 4,5,6| (없음)| R10 도메인 축 + R3(신규 도메인)| 2d|
| 3| BE 가맹점 도메인| BE| be-merchant| 7,8| (없음)| R10 도메인 축 + R3(신규 도메인)| 2d|
| 4| FE 결제 화면| FE| fe-payment| 9| payment/edit.html| R9 + R10| 1.5d|
| 5| FE 가맹점 목록| FE| fe-merchant-list| 10| merchant/list.html| R9 + R10| 1d|
| 6| FE 가맹점 상세 + 정산| FE| fe-merchant-detail| 11,12| merchant/detail.html| R9 + R10| 1.5d|

> **도메인 축 선택 이유 (분리 불가 확정 후)**: 과업 분리 불가 판정 후 적용 — 신규 도메인 2개(결제·가맹점)가 서로 독립 + 인프라가 1.5d로 단일 페이즈 OK → 분리 불가 케이스에서는 도메인 축이 검수 단위 명확. BE 페이즈 2(`be-payment`) 검수 = 결제 API 동작 OK / 페이즈 3(`be-merchant`) 검수 = 가맹점 API 동작 OK. 각 페이즈가 자체 완결 검수 단위. 페이즈 2와 3은 `be-infra` 부모만 공유 → 직접 의존 없음.

> **다시 강조 — 표준 흐름**: 운영상 결제·가맹점은 별도 과업으로 들어오는 게 표준. 묶여 들어오면 4.5단계에서 우선 *과업 분리 권유*. 위 6페이즈 패턴은 분리 불가 예외 케이스 안전망.

**예시 3 — 풀스택 소형 과업 (R9 + R10 동시 적용 — Spring Boot Web / 화면 1개) 농민수당 8888 실 사례 (case study — 다른 프로젝트는 "BE 1 + FE 1 = 2페이즈" 패턴만 차용):**

브리프 §11 14단계 + §7 화면 맵(단일 마법사 화면 1개) → **R9 + R10 동시 적용** → **BE 단일 + FE 단일 = 2 페이즈**:

| N| 페이즈명| 영역| slug| §11 단계| 포함 화면| 적용 규칙| 기간|
|---|---------|------|------|---------|---------|----------|------|
| 1| BE 인프라 & API & Generalize| BE| be| 1,2,3,4,5,8,9,10| (없음)| R1 + R10 단일 BE| 2d|
| 2| FE 경조사 신청 화면| FE| fe-ceremony-apply| 6,7,13,14| ceremony/apply.html| R9 + R10| 1.5d|

> **R10 단일 BE 패턴**: BE 예상 기간 ≤ 3d면 BE 다단계 분할 불필요. 슬러그는 단순히 `be`. FE 화면이 1개라 FE 페이즈도 1개(`fe-{화면-slug}`).

**예시 4 — REST API 소규모 기능 추가 (R9·R10 미적용 — 비풀스택):**

브리프 §11 5단계, 프로젝트 유형 = REST API → **R9·R10 미적용** (화면 없음, 풀스택 아님).

| N| 페이즈명| §11 단계| 적용 규칙| 기간|
|---|---------|---------|----------|------|
| 1| DTO·Repository·Service| 1,2| R1(공통 인프라)| 1d|
| 2| Controller + API 스펙| 3| R4| 0.5d|
| 3| 배포 준비| 4| R5(보정 — 테스트 페이즈 폐지)| 0.5d|

> **분할 근거 §5-2 명시 예시**: "R9·R10 미적용 — REST API 마이크로서비스, 화면 없음. R1·R4·R5만 적용. 페이즈 §DoD·§검증 방법 모두 미생성, 페이즈 종료 신호 = `./gradlew :{module}:test` GREEN (DoD 폐기)."

**예시 5 — Batch Job (R9·R10 미적용):**

브리프 §11 6단계, 프로젝트 유형 = Batch → **R9·R10 미적용**.

| N| 페이즈명| §11 단계| 적용 규칙| 기간|
|---|---------|---------|----------|------|
| 1| Job 스캐폴딩 + Reader| 1,2| R1| 1.5d|
| 2| Processor + Writer| 3,4| (도메인 묶음)| 1.5d|
| 3| 스케줄러 등록 + 마무리| 5,6| R5(보정)| 1d|

> **분할 근거**: 화면이 없으므로 R9·R10 미적용. Reader-Processor-Writer 패턴 단계별 분할.
