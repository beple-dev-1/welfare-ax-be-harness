# Playbook — Phase Partitioning Heuristics (R1~R10)

> dev-planner 4단계 ~ 4.5단계 (페이즈 분할 + 사용자 검토 게이트) 전용 단일 출처.

## 4단계: 페이즈 자동 분할 (휴리스틱 적용)

브리프 §11 구현 순서(일반적으로 10~20단계)를 **페이즈(권장 4~7개, 허용 2~10개)로 재그룹**한다. 정확한 범위는 아래 R7 규칙을 따른다. **풀스택 프로젝트는 R10이 1차 분할 축**(BE/FE 영역 분리), R9는 FE 페이즈 분할 축으로만. 비풀스택은 R1~R8만 적용.

**휴리스틱 규칙 (우선순위 순):**

| 규칙 번호| 규칙| 근거|
|:--:|------|------|
| R1| **스캐폴딩 + 공통 인프라 + 보안 모듈** → Phase 1로 묶음| 밀접 의존, 독립 배포 불가|
| R2| **같은 도메인이어도 v1/v2 버전 분리** → 별도 페이즈| 점진 배포, 독립 검증|
| R3| **신규 도메인(브리프 §3-4에서 '신규' 표기된 기능)** → 독립 페이즈| 리스크 격리|
| R4| **예상 기간 < 0.5영업일 원자 단위** → 앞/뒤 페이즈에 흡수| 페이즈 수 과다 방지|
| R5| **부수 작업 + Deprecate** → 마지막 페이즈에 통합. **테스트·검증·회귀·커버리지·통과 기준 모두 qa-plan 전담**. dev-plan 페이즈는 §DoD·§검증 방법 섹션 모두 만들지 않음 (DoD 폐기 — 페이즈 종료 신호는 BE = `./gradlew :{module}:test` GREEN, FE = sub-agent 코드 생성 완료 보고).| dev-plan(구현 명세) vs qa-plan(테스트 계획) 영역 완전 분리|
| R6| **한 페이즈 예상 기간 1~3영업일 권장**| 점진 배포 가능 규모|
| R7| **권장 4~7 페이즈, 허용 2~10 페이즈** (브리프 §11이 3단계 이하인 초소형 과업은 2페이즈도 허용. 8 페이즈 초과 시 R6 위반 가능성 재검토. R10 적용 시 BE 페이즈 1~2 + FE 화면 페이즈 N개로 N+1~N+2가 자연)| 관리 가능 범위|
| R8| **페이즈 명명: `phase-{N}-{kebab-slug}`**. 슬러그 컨벤션 — 단일 BE: `be` / BE 다단계는 **레이어 축 권장**: `be-{레이어}`(`be-infra`, `be-service`, `be-api`) / **도메인 축은 비권장** (한 과업에 도메인 ≥ 2개 묶여 들어온 예외 케이스 한정 — 우선 과업 분리 검토): `be-infra` + `be-{도메인}`(`be-payment`, `be-merchant`) / 단일 FE: `fe` / FE 화면별: `fe-{화면-slug}`(`fe-login`, `fe-ceremony-apply`) / 비풀스택(R10 미적용): 기존 슬러그(`infra`, `auth`, `entry-v1`)| 파일/브랜치 네이밍 일관성 + 영역·도메인 식별|
| **R9**| **화면 중심 프로젝트는 "화면 1개 = 1페이즈" 단일 원칙 적용** (Web MVC / Spring Boot Web 한정). 횡단 인프라는 R1 우선 유지. 화면 없는 프로젝트는 R9 미적용 명시 후 R1~R8만으로 분할. **R10 채택 시 R9는 FE 페이즈 분할 축으로만 적용** (BE+FE 한 페이즈 묶음 금지). 화면 그룹핑·마법사 단계별 분할 등 묶기 패턴은 적용하지 않는다 — 화면 = 검수 단위.| 사용자 검수 단위와 페이즈 단위 정렬|
| **R10**| **풀스택 프로젝트는 BE/FE 영역 분리를 1차 축으로 적용** (Web MVC / Spring Boot Web 한정). BE는 작업 흐름 묶음(단일 페이즈 또는 다단계 — **레이어 축**(인프라/Service/API) **권장**. **도메인 축**(인프라+도메인별)은 **비권장** — 한 과업에 도메인 ≥ 2개 묶여 들어온 예외 케이스 한정. 운영상 도메인별 과업은 분리되는 것이 표준이므로 도메인 ≥ 2개 케이스에서는 **우선 과업 분리 검토**), FE는 화면별 페이즈. BE 완성 → FE 화면 작업 = 개발 자연 흐름. 한 페이즈에 BE+FE 혼재 금지.| API 변경 위험 차단 + FE 점진 작업 흐름 + 컨텍스트 스위칭 회피|

**작업 산출:** 페이즈 목록 테이블 (루트 §5에 그대로 기록):

```
| N| 페이즈명| slug| 포함 브리프 §11 단계| 예상 기간| 의존 페이즈| 독립 배포 가능|
|---|---------|------|------------------|----------|-----------|---------------|
| 1| 인프라 & 공통| infra| 1, 2, 3| 3d| —| Y|
| 2| Auth & Account| auth| 4, 5| 2d| 1| Y|
| ...
```

분할 근거(어떤 규칙이 어떤 결정에 쓰였는지)를 루트 §5 주석 테이블로 1~2줄씩 기록한다.

### R9 적용 가이드 (화면 중심 프로젝트)

> **R10과의 관계**: 풀스택 프로젝트(Web MVC / Spring Boot Web)는 **R10이 1차 분할 축**이고, R9는 **FE 페이즈 분할 축으로만** 사용된다. BE+FE를 한 페이즈에 묶는 것은 R10에 위배. 비풀스택 프로젝트(R10 미적용 케이스는 없음 — R10이 풀스택 한정 규칙이므로)에서는 R9 단독 적용. 실제로 R9 적용 대상 = R10 적용 대상이지만, 시각이 다름 — R9는 "사용자 가치(화면)", R10은 "개발 작업 흐름(BE/FE)".

**적용 대상 프로젝트 유형:**
- Web MVC WAR (`{프로젝트명}`) — R10과 함께 적용 (R9는 FE 페이즈 분할에만)
- Spring Boot Web (`{프로젝트명}`) — R10과 함께 적용
- REST API 마이크로서비스 (`{프로젝트명}`)
- Batch / Daemon (`{프로젝트명}`)
- Proxy / Router (`{프로젝트명}`)
- Library (`{프로젝트명}`)
- 단일 서비스 (`{프로젝트명}`)

**판정 방법:** 브리프 §2 `프로젝트 유형` 필드 + 브리프 §7(화면 맵) 존재 여부.
- 프로젝트 유형이 Web MVC/Spring Boot Web + §7에 화면 트리 존재 → **R9 + R10 동시 적용** (R9는 FE 페이즈 분할 축, R10은 BE/FE 영역 분리 축)
- 그 외 → **R9·R10 모두 미적용** (분할 근거 §5-2에 "R9·R10 미적용 — 비풀스택 프로젝트" 명시. R1~R8만 사용)

**화면 분할 원칙 (R9 적용 시):**

- **화면 1개 = 1페이즈** 단일 원칙. 화면을 묶지 않는다.
- **횡단 화면 예외 1건만** — 로그인·OTP·세션 만료 같은 인증 횡단 화면은 R1(인프라) 또는 별도 인증 페이즈(`be-auth` / `fe-login`)에 흡수.
- **신규 도메인은 R3 우선** — 신규 도메인 화면은 도메인 단위로 독립 페이즈. R9 적용 대상에서도 R3가 R9보다 우선.

> **이유**: 페이즈 = 사용자 검수 단위. 화면이 검수 단위인데 묶으면 검수 분리가 깨진다. 화면 1개가 너무 커서(3d 초과) 분할이 필요하다는 판단이 들면 그건 **화면 자체를 더 잘게 정의**해야 하는 신호이지 페이즈 분할 옵션이 아니다 (UI 흐름 분할은 dev-plan 영역 밖).

**화면 컬럼 추가 (4단계 작업 산출 + 4.5단계 검토 게이트):**

R9 적용 프로젝트는 페이즈 목록 테이블에 **포함 화면** 컬럼을 추가한다:

```
| N| 페이즈명| slug| 포함 §11 단계| 포함 화면| 예상 기간| 의존 페이즈| 독립 배포|
|---|---------|------|------------|---------|----------|-----------|---------|
| 1| 인프라 & 공통 & 보안| infra| 1,2,3| (없음 — 횡단)| 3d| —| Y|
| 2| 로그인| auth| 4,5| login.html, otp-modal| 1.5d| 1| Y|
| 3| 도메인 A 관리| domain-a| 6,10| domain-a/list, domain-a/detail| 2d| 1| Y|
| 4| 도메인 B v1| domain-b-v1| 7,11| domain-b/v1/step1~3| 2.5d| 2| Y|
| 5| 도메인 B v2| domain-b-v2| 8| domain-b/v2/step1~3| 2d| 4| Y|
| 6| 외부 시스템 연계| external-link| 9| external/link| 1.5d| 2| Y|
| 7| 도메인 C (신규) + 마무리| domain-c| 12,13,14,15| domain-c/*| 3d| 3,5,6| Y|
```

R9 미적용 프로젝트는 **포함 화면** 컬럼을 생략하거나 `해당 없음 — 화면 없는 프로젝트`로 단일 행 처리한다.

**4.5단계 사용자 검토 게이트에서의 활용:**

R9 적용 시 검토 게이트 텍스트는 화면 단위 페이즈 분할 결과 중심으로 출력 (구체 포맷은 4.5단계 본문 참조):
- 페이즈 테이블에 "포함 화면" 컬럼 표시

### R10 적용 가이드 (BE/FE 영역 분리 — 풀스택 프로젝트)

> **핵심 시각**: 개발 자연 흐름은 BE 완성 → FE 화면 작업이다. 한 페이즈에 BE+FE를 섞으면 컨텍스트 스위칭 비효율 + API 변경 시 FE 재작업 위험 + 페이즈 단위가 BE 작업 흐름과 FE 화면 단위 사이에서 모호해진다. R10은 **개발 작업 흐름 단위**로 페이즈를 나눠 자연 흐름과 정렬한다.

**적용 대상 프로젝트 유형 (R9와 동일 — 풀스택 한정):**
- Web MVC WAR — BE만 있음이 아닌 풀스택
- Spring Boot Web — 풀스택
- REST API 마이크로서비스 — BE만 있음, R10 무의미
- Batch / Daemon — 화면 없음, R10 무의미
- Proxy / Router — 화면 없음, R10 무의미
- Library — 화면 없음, R10 무의미
- 단일 서비스 — 화면 없음, R10 무의미

**판정 방법:** 브리프 §2 `프로젝트 유형` + §7(화면 맵). R9 판정과 동일.
- 풀스택 프로젝트 → **R10 적용 (BE/FE 영역 분리)**
- 그 외 → **R10 미적용** (분할 근거 §5-2에 "R10 미적용 — 비풀스택" 명시)

**BE/FE 분할 패턴:**

| 패턴| 구조| 슬러그|
|-----|-----|------|
| **단일 BE + 단일 FE**| Phase 1: `be` (BE 통합) → Phase 2: `fe` (FE 화면)| `be` / `fe`|
| **단일 BE + 화면별 FE** (화면 ≥ 2개)| Phase 1: `be` (BE 통합) → Phase 2~N: `fe-{화면}` 화면별 FE| `be` / `fe-login` / `fe-apply` ...|
| **BE 다단계 (레이어 축) + 화면별 FE** **권장** (BE > 3d, 인프라 비중 큼 + 도메인 1개)| Phase 1~M: `be-{레이어}` (인프라 → Service → API) → Phase M+1~N: `fe-{화면}` 화면별 FE| `be-infra` / `be-service` / `be-api` / `fe-{화면}` ...|
| **BE 다단계 (도메인 축) + 화면별 FE** ️ **비권장 — 예외 케이스 한정** (한 과업에 도메인 ≥ 2개 묶여 들어와 분리 불가한 경우. 표준은 과업 분리.)| Phase 1: `be-infra` → Phase 2~M: `be-{도메인}` (도메인별 자체 완결) → Phase M+1~N: `fe-{화면}` 화면별 FE| `be-infra` / `be-payment` / `be-merchant` / `fe-{화면}` ...|
| **BE만 변경 (FE 영향 없음)**| Phase 1~M: `be-{레이어}` (권장) 또는 `be-infra` + `be-{도메인}`(비권장 — 예외 케이스 한정)| `be-infra` / `be-service` ... 또는 `be-infra` / `be-payment` ...|
| **FE만 변경 (BE 영향 없음)**| Phase 1~N: `fe-{화면}` 화면별. BE 페이즈 없음| `fe-{화면}` ...|

**BE 다단계 분할 기준 (1) — 기간:**
- BE 단일 페이즈 예상 기간 ≤ 3d → **단일 BE 페이즈** (`be`)
- 3d < BE ≤ 6d → **BE 2단계 분할** 검토
- BE > 6d → **BE 3단계 이상 분할** 권장
- 분할 근거: R6(페이즈당 1~3d) 준수 + 의존성 명확화

**BE 다단계 분할 기준 (2) — 축 선택 (레이어 축 vs 도메인 축):**

> **운영 정책 (선결 검토)**: 한 과업에 도메인 ≥ 2개가 묶여 들어왔다면 **우선 과업 분리를 검토**한다. 운영상 도메인별 과업은 분리되는 것이 표준 — 분리 가능하면 분리 후 각 과업당 도메인 1개 = 레이어 축 또는 단일 BE로 자연 처리. 도메인 축은 **분리 불가한 예외 케이스 한정** (운영상 드문 시나리오).

| 상황| 권장 축| 이유|
|-----|--------|------|
| **도메인 ≥ 2개 묶여 들어옴 (선결)**| **우선 과업 분리 검토** → 분리 후 도메인 1개씩| 운영 표준. 분리 가능하면 도메인 축 자체 회피|
| 인프라(공통 설정·마이그레이션·ServiceCore generalize 등) ≥ 1d + 도메인 1개| **레이어 축** 권장 (`be-infra` / `be-service` / `be-api`)| 인프라가 큰 비중을 차지하므로 레이어로 분리하는 것이 자연|
| 도메인 ≥ 2개 + 분리 불가 (예외 케이스)| **도메인 축** ️ 비권장 (`be-infra` + `be-{도메인}` × N)| 페이즈 검수 단위 = 그 도메인 API 동작 OK (자체 완결). **선결 검토에서 분리 불가 판정 후에만 적용**|
| 인프라 큼 + 도메인 여러 개 + 분리 불가 (예외 케이스)| **도메인 축 우선** ️ 비권장 (`be-infra` + `be-{도메인}` × N) — 도메인 내부 레이어 직렬은 페이즈 내 Task로 자연 발생| 검수 단위 명확성 우선. **선결 검토 후에만 적용**|
| 인프라 작음(< 0.5d) + 도메인 1개| **단일 BE** (`be`, R4 흡수)| 분할 비용 > 이득|

> **검수 단위 관점**: 레이어 축은 `be-service` 검수 시 단위 테스트만 / `be-api` 도달해야 진짜 동작 확인. 도메인 축은 페이즈 검수 = 그 도메인 API 동작 OK = 자체 완결. 검수 단위 명확성으로만 보면 도메인 축이 자연 정렬되지만, **운영상 도메인별 과업은 분리되는 게 표준이므로 도메인 축 적용 자체가 비권장**. 도메인 ≥ 2개가 한 과업에 묶여 분리 불가한 예외 케이스 한정 안전망으로 유지.

> **혼재 금지**: 한 과업의 BE 다단계 분할은 한 축으로 통일 (레이어 축 또는 도메인 축 중 하나). 혼재(예: Phase 2 = `be-service`, Phase 3 = `be-payment`)는 일관성 깨짐. 단, `be-infra`는 두 축 공통 진입점 — 도메인 축에서 `be-infra` + `be-{도메인}` × N 조합은 정상.

**FE 화면 분할 (R9 적용):**
- R10 적용 시 R9는 **FE 페이즈 분할 축으로만** 적용 — **FE 화면이 N개면 페이즈도 정확히 N개** (화면 1:1 단일 원칙, 그룹핑 금지)
- 화면이 1개뿐이면 단일 FE 페이즈 (`fe` 또는 `fe-{단일 화면 slug}`)

**의존성 패턴:**
```
BE 단일 + FE 단일:
 Phase 1 (be) → Phase 2 (fe)

BE 단일 + FE 화면별 (화면 3개):
 Phase 1 (be) → Phase 2 (fe-login)
 ↘ Phase 3 (fe-apply)
 ↘ Phase 4 (fe-history)

BE 다단계 (레이어 축) + FE 화면별:
 Phase 1 (be-infra) → Phase 2 (be-service) → Phase 3 (be-api)
 → Phase 4 (fe-login)
 → Phase 5 (fe-apply)

BE 다단계 (도메인 축) + FE 화면별: ️ 비권장 — 예외 케이스 한정 (도메인 ≥ 2개 묶여 분리 불가 시)
 Phase 1 (be-infra) → Phase 2 (be-payment) → Phase 4 (fe-payment)
 → Phase 3 (be-merchant) → Phase 5 (fe-merchant)
```

**4단계 작업 산출 + 4.5단계 검토 게이트에서의 활용:**

R10 적용 프로젝트는 페이즈 목록 테이블에 **영역(BE/FE)** 컬럼을 추가한다:

## 4.5단계: 페이즈 분할 검토 (사용자 게이트)

4단계에서 분할한 페이즈 목록을 **사용자에게 텍스트로 제시하고 확인 받는다.** 5단계 코드 탐색 진입 전에 분할 자체를 검증하는 단계. 페이즈 분할은 develop·qa-test가 묶이는 가장 큰 의사결정이므로, 잘못된 분할로 5~7단계가 모두 낭비되는 것을 차단한다.

**선결 검토 — 도메인 ≥ 2개 묶임 케이스 (필수):**

브리프 §3-4 영향 도메인 또는 §11 단계 분포에서 **도메인 ≥ 2개**가 한 과업에 묶여 있고 BE 다단계 분할 후보가 도메인 축으로 향하는 경우, 페이즈 테이블 출력 직전에 사용자에게 **과업 분리 가능 여부**를 먼저 확인한다.

```
️ 선결 검토: 도메인 ≥ 2개 묶임 감지

이 과업에는 도메인 N개가 묶여 있습니다: {도메인 A}, {도메인 B}, ...
운영 표준은 **도메인별 과업 분리**입니다 — 분리 가능하면 각 과업당 도메인 1개로 처리하는 것이 표준 흐름입니다.

선택:
- "분리 가능" — 과업을 도메인별로 분리하시겠습니까? (dev-interview 재요청 권유)
- "분리 불가" — 도메인 축 (`be-infra` + `be-{도메인}` × N) 으로 분할 진행 (예외 케이스 안전망)
- 사유 없이 "진행" — 사용자 의사 확인 1회 더 요청
```

**사용자 응답 분기:**
- "분리 가능" / "분리하겠다" → dev-planner 종료 권유 + dev-interview 재요청 안내. 사용자가 분리 후 신규 과업으로 dev-plan 재호출.
- "분리 불가" / "묶어서 진행" → 도메인 축 분할 진행 + 페이즈 테이블 출력 (아래 R9+R10 포맷)
