# Large Diff Policy — 대용량 변경 처리 (전체 리뷰 보장)

> code-review skill / code-reviewer agent 의 대용량 diff 처리 규격 **단일 출처**.
> 리뷰는 격리 컨텍스트(code-reviewer 에이전트)에서 수행되므로, 큰 diff 라도 **전부** 리뷰하는 것이 위임의 목적이다.

## 핵심 원칙 — 전체 리뷰

- **하드 파일 캡 없음. per-file truncation 없음.** 변경된 **모든 파일·모든 라인**을 리뷰한다.
- 파일 수·라인 수가 많다는 이유로 파일을 **드롭하지 않는다**.

## 우선순위 = 리뷰/출력 순서 (드롭 아님)

대용량 diff 는 아래 순서로 리뷰·출력하여 중요한 변경을 먼저 다룬다. 순서일 뿐, 뒤 순위 파일도 **반드시 리뷰한다**.

| 순서 | 파일 유형                           | 이유                               |
| ---- | ----------------------------------- | ---------------------------------- |
| 1    | Java 소스 (`.java`)                 | 비즈니스 로직, 버그·보안 영향 최대 |
| 2    | 쿼리 (`.sql`, JPQL)                 | 쿼리 성능·보안 영향                |
| 3    | 프론트엔드 (`.js`, `.html`, `.css`) | XSS·UI 이슈                        |
| 4    | 빌드 설정 (`build.gradle.kts`)      | 의존성 충돌·버전 관리              |
| 5    | 기타                                | 설정, 리소스 등                    |

동일 순위 내에서는 변경 라인 수가 많은 파일을 먼저 다룬다.

## 컨텍스트 초과 시 — 배치 분할 (silent drop 금지)

단일 패스로 컨텍스트에 담기 어려운 초대용량 diff 는 다음으로 처리한다:

1. 프로젝트/파일 유형별로 **배치 분할**하여 다회 패스로 리뷰한다.
2. 각 패스의 발견 이슈를 **합산**하여 단일 마크다운(헤더 이슈 수 집계 `🔴/🟡/🔵`)으로 출력한다.
3. 어떤 파일도 **silent drop 하지 않는다**. 불가피하게 생략된 파일이 있으면 출력 말미에 생략 파일 목록을 **명시**한다.
