# /dev-interview 스킬

기획서 경로 또는 개발 주제를 입력받아 1:1 Q&A를 통해 요구사항을 구체화하고 개발 브리프를 작성한다.

## 사용법
```
/dev-interview [기획서 경로 | 개발 주제]
```

## 절차

### 1단계: 기본 정보 수집
- 과업 번호, 개발 주제, 관련 도메인(ceremony/member/merchant/common/batch)을 확인한다.
- 기획서 파일이 제공된 경우: PDF/PPTX/DOCX 파일이면 parse-spec-doc 스킬로 텍스트 변환 후 읽는다.
- 기획서가 없는 경우: 개발 주제만으로 진행한다.

### 2단계: 병렬 선탐색 (3개 에이전트 동시 실행)
다음 3개 에이전트를 동시에 실행하여 기존 코드와 DB 구조를 파악한다:
- **code-investigator**: `src/main/java/com/beplepay/weadk/welfare` 기준 유사 구현 탐색
- **db-meta-manager**: PostgreSQL에서 관련 테이블·컬럼·인덱스 메타 수집
- **security-auditor**: 보안 관련 코드(인증, 암호화, PII 처리) 탐색

### 3단계: 보완 질문
선탐색 결과와 기획서를 바탕으로 불명확한 사항을 질문한다.
질문은 한 번에 최대 5개, 답변을 받으면 추가 질문으로 이어간다.

중점 질문 영역:
- 경조사 유형 (결혼/출산/장례 등) 및 처리 방식 차이
- 지원 한도 및 지급 조건
- 신청→승인→지급 워크플로우
- 중복 신청 방지 요구사항
- 기존 시스템 연동 여부
- 공통으로 사용할 수 있는 기능 vs 경조사 전용 기능 구분

### 4단계: 개발 브리프 작성
`target/works/{과업번호}_dev_brief.md` 경로에 저장한다.

브리프에 포함할 11개 항목:
1. 과업 개요 (목적, 배경)
2. 대상 도메인 (ceremony/member/merchant/common)
3. 기능 요구사항 목록
4. 비기능 요구사항 (성능, 보안, 가용성)
5. 대상 API/화면 목록
6. DB 변경 사항 (테이블/컬럼 추가·변경)
7. 연동 대상 (외부 API, 내부 서비스)
8. 보안 고려사항
9. 테스트 기준
10. 산출물 위치
11. 미결 사항 (답변 대기 중인 항목)

부록: 질의응답 로그

### 5단계: 누락 검토
브리프 완성 후 다음 항목이 빠지지 않았는지 자가점검:
- [ ] 경조사 지급·취소 시 중복 처리 방지 기준 명시
- [ ] 지원 한도 및 잔액 검증 로직 기준 명시
- [ ] 트랜잭션 경계 (어디까지 한 트랜잭션인지)
- [ ] 권한·접근 제어 기준
- [ ] 공통(common) vs 경조사 전용(ceremony) 분리 계획
- [ ] 배치 처리 시 재처리·실패 처리 기준 (해당하는 경우)

## 완료 후
브리프 저장 경로를 안내하고 `/dev-plan {과업번호}` 명령으로 계획서를 작성하도록 안내한다.
