# 11섹션 개발 브리프 스키마 (출력 계약)

`dev-interview` 가 인터뷰 완료 후 `target/works/{과업번호}_dev_brief.md` 에 저장하는 마크다운 출력 계약.

> 빈 섹션은 **"해당 없음"** 명시 — 임의 추론·과대 채움 금지.
> §1-§11 번호 체계 불변 (`dev-plan` / `dev-planner` 가 명시 파싱하므로 변경 금지).

---

## 스키마 본문

```markdown
# 개발 브리프 — {과업번호} {주제}

## 1. 문서 메타
| 항목 | 값 |
|------|-----|
| 기획서 파일 | {경로 또는 "없음 (인터뷰만)"} |
| 기획서 버전 | {버전 또는 "-"} |
| 기획서 품질 등급 | {자율 평가 — HIGH/MEDIUM/LOW 또는 "해당 없음"} |
| 인터뷰 일자 | {yyyy-MM-dd} |
| 과업번호 | {n} |
| 작업 브랜치 | {브랜치 또는 "미정"} |
| 제안 개발 기간 | {n 영업일} |

## 2. 시스템 결정
| 항목 | 결정 | 근거 |
|------|------|------|
| 개발 유형 | 신규 | 기존 확장 (brownfield) | {근거} |
| 프로젝트 유형 | REST API | Web MVC | Spring Boot Web | Batch | ... | {근거} |
| 프로젝트명 | {이름} | {명명 규칙} |
| 베이스 패키지 | {{config.basePackagePattern}} (예: com.acme.platform.{xxx}) | 공통 규칙 |
| 배포 포맷 | JAR | WAR | {근거} |
| 빌드 | Maven {단일|멀티모듈} | {근거} |

## 3. 타겟 스코프
### 3-1. Primary (쓰기 대상)
### 3-2. Related — 호출 대상 (읽기·참조)
### 3-3. Related — 패턴 참조
### 3-4. 영향 테이블 (신규 포함)

## 4. 업무↔코드 매핑
### 4-1. 기획서 번호 체계 → 내부 API URI
### 4-2. 기획서 단계 → 엔드포인트 (제안)
### 4-3. 기획서 메뉴 → 컨트롤러 (제안)
### 4-4. 기획서 용어 ↔ DB 컬럼

## 5. 코드값 사전
### 5-1. 상태·구분 코드 (DB column_comment 자동 수집)
### 5-2. 기획서 표현 → 코드 매핑 (역방향)

## 6. 외부 통신 보안 스펙
(API 호출 헤더/암호화/서명, SSO, 세션 등 — 없으면 "해당 없음")

## 7. 화면 맵
(템플릿/JS 경로 트리 — 없으면 "해당 없음")

## 8. 특수 요구사항
### 8-1. 보안 (기획서 미명시 보안 요구)
### 8-2. 인프라 (프로젝트 유형별 기본 탑재)
### 8-3. UI 프레임워크
### 8-4. 알림/이벤트 연계

## 9. 미결사항 (Open Questions)
(담당자·별첨문서 확인 필요 항목, 또는 인터뷰 중 보류된 갭)

## 10. 재사용 가능 자산
### 10-1. 라이브러리 ({{config.commonUtilsArtifact}} 등 — project.yaml `commonUtilsArtifact` 단일 출처)
### 10-2. 참조 코드 패턴
### 10-3. 공통 모듈·설정

## 11. 구현 순서 제안
(의존성 기반 순번, Phase 단위 권장)

---

**선탐색 시간**: ~분
**인터뷰 라운드**: n회 (1:1 대화형)
**자동 매핑 커버리지**: ~%
**Codex 검토 등급**: GREEN | YELLOW (메모: ...) | RED (escalate)
```

## Stage 1 형식 게이트 (self-check)

브리프 임시 작성 후 dev-interview 본인이 다음 체크리스트 수행:

- [ ] §1 7항목 비어있지 않음 (작업 브랜치 "미정"은 OK)
- [ ] §2 6항목 결정 + 각 항목 근거 1줄 이상
- [ ] §3 Primary ≥ 1, 영향 테이블 ≥ 1 (또는 정당한 "해당 없음 + 사유")
- [ ] §9 빈 칸 아님 ("없음" 또는 갭 리스트)
- [ ] §11 Phase ≥ 3단계 + 의존성 명시
- [ ] 프로젝트 유형 vs §7 일관성 (Web MVC / Spring Boot Web 이면 §7 채워짐)
- [ ] "TBD" / "추후 결정" 표현 없음 (§1 작업 브랜치 "미정" 제외)

실패 시 본인이 보정 후 재실행.
