## 갭 우선순위 (자율 판단)

1. **스코프·프로젝트 결정** — 이후 모든 결정의 전제
2. **보안·외부 연동** — 의존성 긴 것 먼저
3. **기획서 모호 해석** — 업무 플로우 확정
4. **UI/UX 세부**
5. **운영 정책**

## 갭 카테고리 4종

| 카테고리 | 의미 | 해결 경로 |
|---------|------|----------|
| A. 외부 스펙 | 별첨 문서 필요 | 별첨 요청 |
| B. 설계 결정 | 개발자 판단 | 검증 질문 |
| C. 코드값 미정 | DB 미존재 | DB 설계 |
| D. 기획서 모호 | 해석 다수 | 사업부 확인 |

> **A 카테고리 오분류 방지**: A로 분류 전 sub-agent `code-investigator` 결과 재확인. 기존 컨트롤러가 곧 스펙이면 A 갭 아님.

## 라운드 메타 누적 (Q&A 로그 부록 소스)

각 라운드 종료 시 다음 메타를 **내부 변수에 push**:

| 필드 | 형식 |
|------|------|
| `#` | 라운드 번호 (1부터) |
| `카테고리` | A / B / C / D |
| `강도` | STRONG / MEDIUM / WEAK |
| `질문요약` | ≤80자 |
| `응답요약` | ≤80자, 결정 사실만, **원문 인용 금지** |
| `반영§` | §X-Y (재인터뷰면 `§X (재인터뷰 R{n})`) |

→ Stage 2 통과 후 `templates/qna-log-appendix.md` 형식으로 브리프에 첨부.
