# 1:1 라운드 헤더 포맷

`dev-interview` 가 각 라운드의 질문을 출력할 때 사용하는 템플릿. **전달 방식**은 강도가 아니라 **가설 유무**로 분기한다. ("소크라테스식" 은 질문 태도(열린 탐색·유도 안 함)이지 위젯이 아니다 — `AskUserQuestion` 도 자동 "Other" 자유입력이 붙어 열린 탐색을 보장하므로 동일하게 소크라테스식이다.)

- **가설 있음** (STRONG/MEDIUM, 또는 WEAK 라도 약한 가설이 잡히면) → 네이티브 `AskUserQuestion` + Other (**기본**). 근거·진행률은 호출 직전 텍스트 1줄.
- **가설 전무** → 텍스트 열린 질문 (도구 미사용). 억지 옵션 생성은 앵커링(anchoring) 유발 → 오히려 비-소크라테스이므로, 가설이 전혀 없을 때만 이 경로.

어느 경로든 **한 번에 1질문** (`AskUserQuestion` 호출 시 `questions` 배열에 객체 1개만 — 도구는 4개까지 허용하나 본 스킬은 1개로 고정하여 연쇄 영향·근거 갱신을 보존).

---

## 강도 판단 기준

| 강도 | 판단 | 전달 방식 (= 가설 유무) |
|------|------|----------|
| **STRONG** | 유사 패턴 + DB·기존 구현 명확 | `AskUserQuestion` + Other — 가설 옵션 2~3 + 추천(첫 옵션) |
| **MEDIUM** | 유사 패턴 있으나 근거 약함 | `AskUserQuestion` + Other — 2안 + 추천(첫 옵션) |
| **WEAK** | 근거 빈약 | 약한 가설 잡히면 `AskUserQuestion` + Other (기본) / **가설 전무면** 텍스트 열린 질문 |

---

## STRONG / MEDIUM — AskUserQuestion 가설 기반

도구 호출 **직전**에 진행률·근거를 텍스트 1줄로 노출한 뒤 `AskUserQuestion` 1질문 호출:

```markdown
라운드 {n}/{예상총} — 진행률: {해결된}/{전체} — 강도: STRONG(또는 MEDIUM) — 근거: {유사 패턴 / DB 신호 / 기존 구현 1줄 인용}
```

```jsonc
// AskUserQuestion 호출 — questions 배열에 객체 1개만
{
  "questions": [{
    "header": "{갭 카테고리 칩 ≤12자}",   // 예: "설계 결정" / "외부 스펙" / "코드값" / "기획서 모호"
    "question": "{갭 항목} (근거: {1줄 요약})",
    "multiSelect": false,                  // 복수 선택 본질 갭(예: 지원 채널 다중)만 true
    "options": [
      { "label": "{추천안} (추천)", "description": "{추천 이유 — 트레이드오프 1줄}" },
      { "label": "{옵션2}",         "description": "{트레이드오프 1줄}" },
      { "label": "{옵션3 — 해당 시}", "description": "{트레이드오프 1줄}" }
    ]
  }]
}
```

규약:

- **추천 옵션은 반드시 `options[0]`** 이고 label 끝에 `(추천)` 표기 (도구 규약: 추천은 첫 옵션).
- `header` 는 갭 카테고리 칩 (`gap-categories.md` 4종 매핑). `label` 1~5단어, `description` 트레이드오프 1줄.
- "그 외" 여지는 도구가 **자동 제공하는 "Other" 선택지**가 보장 (별도 옵션 추가 불요).
- `모르겠음/알아서` 분기가 필요하면 명시 옵션 추가 (예: `label: "추천대로(잘 모르겠음)"`) 또는 Other 자유텍스트로 흡수.
- 설계안 코드 스니펫 비교가 필요한 갭은 `options[].preview` 에 스니펫 첨부 (선택적, multiSelect=false 일 때만).

## 가설 전무 — 텍스트 열린 질문 (도구 미사용)

```markdown
라운드 {n}/{예상총} — 진행률: {해결된}/{전체} — 모드: 소크라테스식

**상황**: {확인된 사실 1~2줄} / {불확실한 부분 1줄}
**강도**: WEAK (근거 부족)

**질문**: {열린 질문 — 목적/시나리오/반대관점/경계 중 택1}

답변에 따라 후속 질문 드리겠습니다.
```

> **1질문 가드 (텍스트 경로도 동일)**: **한 번에 1질문만**. 항목이 여러 개여도(예: 적용일·하위호환·기존데이터) 묶지 말 것 — 설계 영향 가장 큰 1개만 묻고 나머지는 다음 라운드로 이연. 번호 매긴 복수 하위질문(1·2·3) 묶음 = 위반.

> **이 텍스트 경로는 가설이 전혀 없을 때만 사용.** 약한 가설이라도 잡히면 위 `AskUserQuestion` + Other 경로가 기본 (억지 객관식이 아니라 진짜 가설 + Other 이므로 앵커링·확증 편향 미발생 — 원칙 #9 충족).

---

## 사용자 응답 처리 규칙

`AskUserQuestion` 은 구조화 응답(`answers`)을 반환 — 정규식 파싱 불요.

| 응답 | 해석 |
|------|------|
| 옵션 선택 (label) | `answers` 의 선택 라벨 직접 사용. 추천 옵션 선택 = 추천 채택 |
| **"Other"** (자유 입력) | 객관식 거부 → 다음 라운드에서 텍스트 열린 질문으로 재질의 |
| `추천대로(잘 모르겠음)` 옵션 | 합리적 기본값 적용 + 1회 확인 |

답변이 다른 갭에 연쇄 영향 → 다음 갭 `options`·근거 갱신.

---

## 정체·확증 편향 가드

- **같은 갭 2라운드 진전 없음**: 시나리오/반대관점/경계 질문으로 전환 (가설 소진 시 텍스트 열린 질문).
- **연속 3라운드 같은 위치 추천**: 자가 점검 ("동일 논리 반복인가?") → **반대 관점 옵션 1개 추가** (옵션 순서 셔플은 추천=`options[0]` 규약과 충돌하므로 금지).

## 라운드 메타 누적 (Q&A 로그 부록 소스)

각 라운드 종료 시 다음 메타를 내부 변수에 누적:

| 필드 | 값 |
|------|-----|
| `#` | 라운드 번호 (1부터) |
| `카테고리` | A. 외부 스펙 / B. 설계 결정 / C. 코드값 미정 / D. 기획서 모호 |
| `강도` | STRONG / MEDIUM / WEAK |
| `질문요약` | ≤80자 |
| `응답요약` | ≤80자, 결정 사실만, **원문 인용 금지** |
| `반영§` | §X-Y (재인터뷰면 `§X (재인터뷰 R{n})`) |

→ Stage 2 통과 후 `templates/qna-log-appendix.md` 형식으로 브리프에 첨부.
