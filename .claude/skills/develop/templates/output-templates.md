# 출력 템플릿

> `/develop` 스킬에서 사용하는 세션 출력 형식 모음.
> 출력 형식 변경 시 이 파일만 수정한다.

---

## 스코프 미존재 시 오류 메시지

스코프 레지스트리(`.claude/config/scope.yaml`)에 일치하는 식별자가 없을 때 출력한다.

```
❌ 유효하지 않은 스코프입니다: '{입력값}'

사용 가능한 스코프 ({총 개수}개):

{scope.yaml 의 groups 를 순회하여 그룹별 entry 의 id 를 콤마 리스트로 출력}
[{groups.A.name}]  {groups.A.scopes[].id 콤마 연결}
[{groups.B.name}]  {groups.B.scopes[].id 콤마 연결}
...
{subScope 가 정의된 entry 는 `{id} {paramName}` 형태도 함께 표시}

사용 예시:
  /develop ceremony
  /develop member
  /develop common
  /develop admin
  /develop batch
  /develop ceremony --ref-read=common              ← 단일 스코프 참조 허용
  /develop ceremony --ref-read=common,domain ← 콤마 리스트로 다중 허용
```

> 메인 Claude 는 `.claude/config/scope.yaml` 을 Read 하여 그룹별 id 를 동적으로 렌더한다. 데이터 변경 시 본 템플릿 수정 불필요.

## 세션 시작 요약 (9단계)

세션 시작 시 출력하는 요약 형식. 주요 컴포넌트, 아키텍처, 규칙, 가이드라인은 내부 로드만 하고 출력하지 않는다.

```
## 🌿 {스코프 식별자} 개발 세션

| 항목 | 내용 |
|------|------|
| 프로젝트 | {projectRoot} ({projectType}) |
| 브랜치 | 🌿 {branch} |
| 허용 범위 | {허용 경로 + 공유 모듈} |
| 참조 범위 (읽기 전용) | {refReadScopes 콤마 연결} |

> `참조 범위` 행은 `{refReadScopes}` 가 비어있지 않을 때만 출력한다.

{6단계에서 계획서를 로드한 경우}

📋 **개발 계획서**
| 항목 | 내용 |
|------|------|
| 계획서 | target/plans/work-{과업번호}_plan.md |
| 수정/생성 파일 | {파일 수}개 |
| 구현 순서 | {계획서의 구현 순서 요약} |

계획서 기반으로 개발을 시작할까요?

{계획서가 없으면}

어떤 기능을 구현할까요?
```

## 스코프 외 접근 차단 (8단계)

**스코프 외 차단 (기본):**

```
🚫 스코프 제한: '{요청 파일}'은 현재 세션의 허용 범위({허용 경로 목록})에 포함되지 않습니다.
→ 참조만 필요하면 `/develop {메인스코프} --ref-read={해당스코프}` 로 재실행하세요.
→ 수정이 필요하면 `/develop {해당스코프}` 로 스코프를 전환하세요.
  (전환 시 현재 스코프의 보안 점검을 먼저 수행합니다.)
```

**참조 스코프 첫 접근 시 배너 (세션당 스코프별 1회):**

```
🔍 참조 스코프 읽기: {참조 스코프 식별자} (쓰기 금지, 참조 목적으로만 사용)
```

**참조 스코프 쓰기 차단:**

```
🚫 참조 스코프 쓰기 금지: '{요청 파일}'은 읽기 전용 참조 스코프({식별자})에 속합니다.
→ 수정이 필요하면 `/develop {해당 스코프}` 로 스코프를 전환하세요.
```

**`--ref-read` 값 검증 실패:**

```
❌ 유효하지 않은 참조 스코프: '{입력값}'

유효한 스코프 목록은 `.claude/config/scope.yaml` 을 참조하세요.
- 최상위 스코프 식별자만 값으로 지정합니다 (예: ceremony, member, common).
- 프로젝트 루트명(we-adk-welfare-user, we-adk-welfare-domain 등)은 받지 않습니다.
- 하위 스코프는 지원하지 않습니다.
```

## 캐시 브랜치 불일치 자동 재캐싱 (7단계)

캐시 frontmatter `branch:` 가 현재 git 브랜치와 다르거나 `branch:` 필드가 없는 구버전 캐시일 때 출력한다.

```
⚠️ 캐시 브랜치 불일치
  캐시 브랜치: {캐시-브랜치 또는 '없음(구버전)'}
  현재 브랜치: {현재-브랜치}
  → 자동으로 전체 재캐싱을 수행합니다.
```

재캐싱 완료 후:

```
📦 캐시 재생성 완료 (branch: {현재-브랜치}, scanned_at: {ISO 8601})
```

## 참조 스코프 동적 변경 확인 (8.5단계)

세션 중 자연어 트리거(`참조 X 추가/제거`, `ref-read 해제` 등)로 참조 스코프를 변경할 때 출력한다.

```
🔍 참조 스코프 변경 확인
  현재: {현재 상태 요약}
  요청: {해석된 의도}
  진행하시겠습니까? (y/n)
```

사용자 승인 후 상태를 갱신하고 1줄 요약을 출력한다:

```
✅ 참조 스코프 갱신: {갱신된 상태 요약}
```

## 스코프 자가 점검 (10단계)

커밋/종료 전 이번 세션의 스코프 준수 여부를 점검하고 출력한다.

```

## 🔐 스코프 자가 점검

| #  | 점검 항목                                              | 판정              | 비고                                |
| -- | ------------------------------------------------------ | ----------------- | ----------------------------------- |
| 1  | 메인 스코프 외 쓰기 시도 여부 (Edit/Write)             | ✅ 준수 / 🔴 위반 | {파일 목록}                         |
| 2  | 허용되지 않은 스코프 파일 읽기 여부 (ref-read 미포함)  | ✅ 준수 / 🔴 위반 | 참조 목적 읽기: N건 ({스코프 목록}) |

---

{위반 항목이 없을 때}
✅ **스코프를 준수했습니다.** `/code-review` → 커밋 후 `/pack` 으로 세션을 마무리하세요.

{위반 항목이 있을 때}
🔴 **위반 사항이 발견되었습니다. 아래 내용을 수정한 후 커밋하세요.**

#### 🔴 위반 상세 및 수정 방법

- **항목 {N}: {위반 내용}**
  - 위반 코드: `{코드 스니펫}`
  - 수정 방법: {구체적 수정 방법}

```
